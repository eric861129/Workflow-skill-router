# Workflow Skill Router v2.0.0-beta.2

## Why V2

Workflow Skill Router plans the smallest verifiable Skill route before Codex starts work. It separates installed metadata from real runtime authority, preserves an explicitly named Skill, reroutes at Phase boundaries, and binds long-running work to Host-owned Goal state without claiming permissions it does not have.

Beta.2 restores the most valuable V1 capability as a safe V2 contract: users can define Personal or Workspace Skill Trees without forking the Plugin. Profiles are strict JSON routing data, never executable prompts, and a match remains `intended-unverified` until Runtime Capability Discovery confirms activation.

## What changed in beta.2

- Personal and Workspace Routing Profiles with deterministic precedence: explicit user Skill, workspace, personal, then built-in routing.
- Packaged personal and workspace examples plus `profile validate`, `install`, `list`, and `preview` commands.
- Current-Phase-only Primary and immediate support selection; future Phase Skills are never activated early.
- Canonical filesystem trust checks that reject junction or symlink escapes and fail before any outside write.
- A migration backfill and legacy digest path that preserve beta.1 explicit Skill plans and idempotent replay.

```powershell
codex plugin marketplace add eric861129/Workflow-skill-router --ref v2.0.0-beta.2
codex plugin add workflow-skill-router@workflow-skill-router
codex plugin list
```

Offline inspection asset: `workflow-skill-router-plugin-v2.0.0-beta.2.zip`.

### Skill-only

Download `workflow-skill-router-skill-v2.0.0-beta.2.zip` and extract its inner `workflow-skill-router/` directory into the Codex Skills directory. Skill-only preserves task-shape routing, Explicit Skill Lock, reporting, and advisory Profile interpretation. Loading fixed local Profile files requires Host filesystem access; it does not provide durable consent, compare-and-swap, drift enforcement, or `hybrid-full` conformance.

## Evidence boundary

The reviewed 36 attempts and 42 model turns belong to `v2.0.0-beta.1` under evaluation contract 2.2.0. They remain valid historical evidence for that runtime candidate, but they do not evaluate Personal Routing Profiles or this beta.2 source revision.

Beta.2 Profile claims are supported by deterministic contract tests, Router Core integration and migration tests, real junction security regressions, MCP tests, package parity, and documentation/site gates. A new real-model evaluation requires separate quota authorization, sanitized evidence review, zero hard violations, and a new `maintainer-attestation`; beta.1 evidence is not relabeled as beta.2 proof.
