# Workflow Skill Router v2.0.2

> Prepared local candidate — not yet published.

## Why V2

Workflow Skill Router is a pre-execution Skill-selection layer for Codex. It
chooses the smallest verifiable route for Single, Phased, or Managed Goal work
without replacing Host permissions, approvals, sandboxing, or production
orchestration.

## Choose your install mode

- **Plugin + MCP:** the future
  `workflow-skill-router-plugin-v2.0.2.zip` contains the bundled local control
  plane and typed runtime readiness.
- **Skill-only:** the future
  `workflow-skill-router-skill-v2.0.2.zip` preserves routing and disclosure
  instructions without durable state or verified Host authority.

These assets do not exist until the candidate passes promotion and the
authorized release completes. The historical beta.1 behavior evidence remains
bounded to **36 attempts** across **42 model turns**; this patch does not claim
new model evidence, verified Skill activation, token reduction, or task-outcome
improvement.

## Canonical Plugin distribution

This patch establishes one canonical development and governance surface:
`eric861129/Workflow-skill-router`. The standalone
`eric861129/workflow-skill-router-plugin` repository is a generated target for
root-level Plugin installation. Runtime code, tests, Skill content, packaging
policy, and fixes originate in the canonical repository.

The deterministic builder produces the target tree from an exact source
revision. The guarded synchronizer accepts only the configured target identity,
refuses dirty or unmanaged target state, and removes only regular files recorded
as previously managed. The trusted source release workflow publishes that tree
only after the frozen-source preflight and source release succeed.

## Scanner, tag, and governance boundary

- The target HOL Plugin Scanner must succeed for the exact generated target
  revision before the workflow creates the corresponding `v*` tag.
- Scanner failure stops tag creation. An existing mismatched tag fails closed
  and is never rewritten.
- The GET-only governance verifier checks that the target is public, uses a
  protected `main` default branch, requires the Scanner check, protects
  `refs/tags/v*`, and grants the dedicated Release App bypass.
- Installing the App, applying rulesets, dispatching the release, and changing
  the target repository are explicitly authorized external operations. They are
  not performed by local verification.
- Do not manually repair the generated target. Fix canonical source or an
  authorized live prerequisite, review a new candidate, and rerun the trusted
  release workflow.
