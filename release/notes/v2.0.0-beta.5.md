# Workflow Skill Router v2.0.0-beta.5 — Router-owned Local Work Loop candidate

This source tree prepares `v2.0.0-beta.5`; it is not yet published. Stable
`latest` remains V1.3.1, and the latest published `latest-v2` installation
snapshot remains `v2.0.0-beta.3` until the reviewed release workflow completes.

Its checked-in `release_lifecycle` is `prepared-local-candidate`. This first
candidate-content commit intentionally does **not** rebind
`release_source_revision`: the later trusted metadata-only promotion commit
must bind that field to this exact reviewed beta.5 candidate SHA, set
`reviewed-attested-publishable`, and leave the candidate contents unchanged.
`CREATE_V2_RELEASE` alone cannot bypass these guards.

## Why V2

V2 prevents a crowded Codex environment from activating every plausible Skill
or MCP tool at once. It chooses the smallest verifiable route for the current
work boundary, preserves a user-named Skill, and fails closed whenever the
runtime lacks the required authority.

## Choose your install mode

- Plugin + MCP provides the prepared Router-owned Local Work Loop, durable
  routing plans, scoped consent transitions, and runtime readiness. The local
  candidate asset name is `workflow-skill-router-plugin-v2.0.0-beta.5.zip`.
- Skill-only preserves routing and Explicit Skill Lock instructions without
  durable MCP state. The local candidate asset name is
  `workflow-skill-router-skill-v2.0.0-beta.5.zip`.

Until this candidate is published, install the immutable `v2.0.0-beta.3`
release instead of assuming these beta.5 asset names exist on GitHub.

## What changed

- **Router-owned Local Work Loop:** the bundled runtime exposes **4 always local-ready + 3 Router-owned conditional-local** operations. The conditional
  tools are `get_next_work`, `record_work_event`, and `evaluate_gate`; they
  work only for a validated Router-owned graph and remain
  `authority_mode=router-local` with `host_transition_authorized=false`.
- **Authority separation:** a Native Goal, verified activation, formal
  evidence, deployment, or production action still requires verified Host
  authority. Conditional-local never means `7/12 local-ready`, and it never
  bypasses Explicit Skill Lock or scoped consent.
- **Host Integration Kit:** the package now includes a capability manifest,
  reference adapter, and conformance suite. The reference adapter is
  development evidence only; it is not production Host authority and does not
  permit a `hybrid-full` claim.
- **Pilot protocol:** twenty local-task slots, a separate verified-Host lane,
  restricted binding manifests, and a semantic-recommender decision gate are
  frozen before execution. No real Pilot task has been executed or counted.
- **Contract continuity:** Contract 2.3.0 remains sealed. The reference-driver validates deterministic protocol composition and artifacts; it does not prove real-model behavior.

## Evidence and publication boundary

The prepared candidate has no beta.5 live model evaluation, completed Pilot,
verified Host Pilot, `maintainer-attestation`, GitHub prerelease, tag, or
channel update. A fresh beta smoke remains 36 attempts and 42 model turns; it
requires explicit quota authorization, an exact source/case/instruction/model
digest, sanitized-evidence review, zero hard violations, and a
`maintainer-attestation`.

Local builders produce deterministic Plugin and Skill-only ZIPs, checksums,
SBOM, and provenance under `dist/release-2.0.0-beta.5/`. These transient files
are verification outputs, not a claim that a GitHub prerelease exists. Verify
the generated bytes against `checksums.sha256`, inspect the SPDX SBOM and
provenance manifest, and run the extracted Plugin and Skill smoke checks before
the later promotion commit.
